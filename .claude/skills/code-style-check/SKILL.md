# 代码风格检测技能

基于项目 `CLAUDE.md` 中定义的编码规范，对全部 Python 文件进行逐项检测，输出 Markdown 报告并自动修复可修复的违规项。

## 检测流程

### 第一步：加载规范

首先读取项目根目录的 `CLAUDE.md`，解析"编码规范"章节中的全部规则，作为检测基准。

### 第二步：收集文件

使用 Glob 收集项目中所有 `.py` 文件，排除以下目录：
- `__pycache__/`
- `.claude/`
- 虚拟环境目录（`venv/`、`.venv/`、`env/`）

### 第三步：逐文件逐项检测

对每个文件按以下维度依次检测。每个维度检测完毕后立即统计违规数。

#### 3.1 文件头部

- [ ] 是否有模块级 docstring（第一行或 `from __future__` 之后）
- [ ] 是否包含 `from __future__ import annotations`（`__init__.py` 除外）
- [ ] 导入顺序是否正确：标准库 → 空行 → 同包模块（相对导入）
- [ ] `main.py` 是否使用绝对导入（`from student_system.xxx`）
- [ ] 模块内文件是否使用相对导入（`from .xxx`）

#### 3.2 命名规范

- [ ] 变量/函数名：`snake_case`
- [ ] 类名：`PascalCase`
- [ ] 常量：`UPPER_SNAKE`
- [ ] 私有成员：单下划线前缀 `_`
- [ ] 枚举成员：`PascalCase`

#### 3.3 Docstring 风格

- [ ] 所有公开函数/方法是否有 docstring
- [ ] Docstring 是否遵循 Google 风格
- [ ] 是否包含 `Args:`、`Returns:`、`Raises:` 小节（如适用）
- [ ] 类是否有 docstring 及 `Attributes:` 小节
- [ ] Docstring 语言是否为中文

#### 3.4 类型注解

- [ ] 所有函数参数是否有类型注解
- [ ] 所有函数返回值是否有类型注解（含 `-> None`）
- [ ] 所有实例属性是否在 `__init__` 中声明类型注解
- [ ] 是否使用 `| None` 而非 `Optional[...]`
- [ ] 复杂字典是否使用 `TypedDict`

#### 3.5 格式化

- [ ] 缩进是否为 4 空格（无 Tab）
- [ ] 行宽是否 ≤ 100 字符
- [ ] 字符串是否统一使用双引号
- [ ] 顶级类/函数之间是否恰好 2 个空行
- [ ] 类内方法之间是否恰好 1 个空行
- [ ] 模块末尾是否有 `__all__` 定义

#### 3.6 错误处理

- [ ] 自定义异常是否继承自统一基类
- [ ] 业务逻辑是否用异常报告错误（而非返回 `False`/`None`）
- [ ] 异常重抛是否使用 `raise ... from e` 保留异常链

#### 3.7 测试文件（仅 `tests/` 目录下）

- [ ] 测试类命名是否为 `Test<Component>` 格式
- [ ] 测试方法命名是否为 `test_<scenario>` 格式
- [ ] 是否使用 `self.assertXxx` 而非裸 `assert`
- [ ] 参数化是否使用 `with self.subTest(...)`
- [ ] 临时文件是否使用 `tempfile.TemporaryDirectory()`
- [ ] 文件末尾是否有 `if __name__ == "__main__": unittest.main()`

### 第四步：生成报告

生成 Markdown 格式报告，结构如下：

```markdown
# 代码风格检测报告

**检测时间**：YYYY-MM-DD HH:MM:SS
**检测范围**：N 个 Python 文件
**违规总数**：X 项（🔴 错误: E, 🟡 警告: W, 🔵 提示: I）

## 违规概览

| 文件 | 🔴 错误 | 🟡 警告 | 🔵 提示 | 合计 |
|------|---------|---------|---------|------|
| ... | ... | ... | ... | ... |

## 详细结果

### <文件路径>

| # | 行号 | 严重度 | 类别 | 问题描述 | 修复建议 |
|---|------|--------|------|---------|---------|
| 1 | ... | ... | ... | ... | ... |

## 规范符合度

- 文件头部: XX%
- 命名规范: XX%
- Docstring: XX%
- 类型注解: XX%
- 格式化: XX%
- 错误处理: XX%
- 模块接口: XX%
```

严重度定义：
- 🔴 **错误**：缺少必要项（如缺少 docstring、缺少类型注解、命名完全错误）
- 🟡 **警告**：格式不规范（如行宽超标、空行数不对、引号不一致）
- 🔵 **提示**：建议改进（如 docstring 可更详细、可添加 `__all__`）

### 第五步：自动修复

报告生成后，询问用户是否需要自动修复。自动修复范围：

| 可自动修复 | 不可自动修复 |
|---|---|
| 添加缺失的 `from __future__ import annotations` | 命名不规范（需人工判断语义） |
| 调整导入顺序 | 缺失 docstring（需人工撰写内容） |
| 修复空行数量 | 缺失类型注解（需理解业务逻辑） |
| 替换 Tab 为 4 空格 | 异常链缺失（需理解调用关系） |
| 将裸 `assert` 替换为 `self.assertXxx` | 缺少 `__all__`（需人工确认公开接口） |
| 统一字符串引号为双引号 | |
| 添加文件末尾 `if __name__ == "__main__"` | |
| 将 `Optional[X]` 替换为 `X \| None` | |

## 备注

- 检测严格依据 `CLAUDE.md` 中的编码规范进行，不添加额外规则
- 自动修复前会显示变更预览，用户确认后执行
- 对于 `__init__.py`，跳过 `from __future__ import annotations` 检查
